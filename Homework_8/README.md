# Домашнее задание №8 — MVP ML-сервиса

Локальный сервис раннего прогнозирования производственного дефекта.

Сервис предоставляет:

* синхронный прогноз для одной случайной единицы продукции;
* асинхронный расчёт для всех строк выбранного производственного дня;
* raw score и калиброванную вероятность дефекта;
* бинарный прогноз риска;
* прогнозный и фактический суточный процент дефекта;
* Top-10% наиболее рискованной продукции;
* историю запусков и результатов в PostgreSQL;
* REST API и HTML/CSS/JavaScript-интерфейс;
* независимое масштабирование ML-воркеров.

Авторизация, облачное развёртывание, Kubernetes, переобучение и production-мониторинг в MVP не входят.

---

## 1. Архитектура

```text
Пользователь
     |
     v
   Nginx :80
   /       \
  /         \
HTML UI     FastAPI
(index.html)   |
          +----+----+
          |         |
     PostgreSQL  RabbitMQ
                    |
               ML workers
                    |
               Model bundle
```

Компоненты:

* **Nginx** — единая точка входа и раздача статического интерфейса;
* **HTML/CSS/JavaScript** — пользовательский интерфейс без отдельного frontend-контейнера;
* **FastAPI** — REST API;
* **PostgreSQL** — история запусков и построчные результаты;
* **RabbitMQ** — очередь суточных задач;
* **ML worker** — отдельный масштабируемый процесс инференса;
* **CatBoost bundle** — модель, калибратор, порог и схема признаков.

Одиночный прогноз выполняется синхронно внутри API.

Суточный расчёт создаёт запись со статусом `queued`, отправляет `run_id` в RabbitMQ и выполняется свободным ML-воркером.

## 3. ML bundle и логика прогноза

`model_bundle.joblib` содержит:

* CatBoost-модель;
* Isotonic-калибратор;
* raw-порог бинарного решения;
* точный список и порядок признаков;
* числовые и категориальные признаки;
* токен пропусков;
* размер Top-K;
* название и версию модели.

Исходная оценка модели:

```python
raw_score = model.predict_proba(X)[:, 1]
```

Калиброванная вероятность:

```python
calibrated_probability = calibrator.predict(raw_score)
```

Бинарный класс:

```python
predicted_class = raw_score >= threshold_raw
```

Суточный процент дефекта:

```python
predicted_daily_rate = mean(calibrated_probability) * 100
```

Таким образом:

* `raw_score` используется для бинарного решения;
* `calibrated_probability` используется как отображаемая вероятность и для суточной агрегации.

## 4. Демонстрационный датасет

`data_mvp_demo.csv` содержит:

* `row_id`;
* `production_time`;
* `production_day`;
* `actual_target`;
* все признаки модели.

Таргет используется только после инференса для демонстрационного сравнения.

`prepare_features()` выбирает исключительно сохранённый список `feature_names`, поэтому:

* таргет не передаётся модели;
* ID не передаётся модели;
* технические даты не передаются модели;
* лишние колонки игнорируются;
* отсутствие обязательного признака вызывает ошибку;
* порядок признаков совпадает с порядком обучения.

## 5. Доменная модель

PostgreSQL содержит три связанные сущности.

### `model_metadata`

Хранит:

* название и версию модели;
* raw-порог;
* количество признаков;
* точный список признаков.

### `prediction_runs`

Хранит:

* тип запуска: `single` или `daily`;
* статус: `queued`, `processing`, `completed`, `failed`;
* производственный день;
* количество строк;
* прогнозный и фактический процент дефекта;
* абсолютную ошибку;
* размер и долю риск-зоны;
* количество Top-K строк;
* идентификатор worker;
* временные метки;
* сообщение об ошибке.

### `prediction_results`

Хранит:

* обезличенный `row_id`;
* `raw_score`;
* калиброванную вероятность;
* бинарный и фактический классы;
* совпадение прогноза с фактом;
* ранг риска;
* признак Top-K.

Полные производственные признаки в PostgreSQL не дублируются.

## 7. Пользовательские сценарии

### Одиночный прогноз

1. Открыть вкладку «Одиночный прогноз».
2. Нажать кнопку выполнения прогноза.
3. Сервис выбирает случайную строку синтетического демонстрационного датасета.
4. Пользователь получает raw score, вероятность, прогноз, факт и совпадение.
5. Запуск и результат сохраняются в PostgreSQL.

### Суточный прогноз

1. Открыть вкладку «Суточный прогноз».
2. Выбрать производственный день.
3. Нажать «Рассчитать прогноз за день».
4. FastAPI создаёт запуск со статусом `queued`.
5. `run_id` отправляется в RabbitMQ.
6. Свободный worker меняет статус на `processing`.
7. После расчёта статус становится `completed` либо `failed`.
8. Интерфейс показывает агрегаты, Top-10% и построчные результаты.

HTML-интерфейс проверяет статус последовательными запросами. Новый запрос создаётся только после завершения предыдущего, а polling прекращается после `completed`, `failed` или ошибки.

## 8. REST API

```text
GET  /api/health
GET  /api/model
GET  /api/data/dates
POST /api/predictions/random-row
POST /api/predictions/day
GET  /api/predictions/runs
GET  /api/predictions/runs/{id}
GET  /api/predictions/runs/{id}/results
```

Тело суточного запроса:

```json
{
  "production_day": "2026-04-10"
}
```

* некорректный формат даты возвращает `422`;
* отсутствующий в данных день возвращает `404`;
* успешное создание асинхронной задачи возвращает `202 Accepted`.

## 9. Тесты

Запуск:

```bash
docker compose run --rm api python -m pytest -q
```

Тестами проверяются:

* загрузка bundle и датасета;
* бинарность и полнота таргета;
* точный состав и порядок признаков;
* отсутствие таргета среди признаков;
* отсутствующий и лишний признак;
* диапазон raw и калиброванных вероятностей;
* применение raw-порога;
* суточная агрегация, риск-зона и Top-K;
* FastAPI endpoints и валидация дат;
* создание queued-запуска;
* сохранение одиночного прогноза;
* успешный и ошибочный сценарии worker;
* сохранение результатов в PostgreSQL.

В unit/integration-тестах публикация сообщения подменяется, а worker вызывается напрямую. Полная цепочка `FastAPI → RabbitMQ → worker` дополнительно проверяется smoke-тестом в запущенном Docker Compose.

## 10. Масштабирование ML-воркеров

По умолчанию работает один worker.

```bash
docker compose up -d --scale worker=3
```

Проверка:

```bash
docker compose ps
```

Все worker-контейнеры:

* загружают собственную копию модели;
* читают общую RabbitMQ-очередь;
* получают разные задачи;
* работают независимо от FastAPI и HTML-интерфейса.

Возврат к одному worker:

```bash
docker compose up -d --scale worker=1
```

## 11. Соответствие критериям задания

1. Доменная модель — `ModelMetadata`, `PredictionRun`, `PredictionResult`.
2. СУБД — PostgreSQL.
3. REST API — FastAPI и Swagger.
4. Пользовательский интерфейс — HTML/CSS/JavaScript через Nginx.
5. Тесты — pytest для данных, модели, API, БД и worker.
6. Docker — полный запуск через Docker Compose.
7. Масштабирование — RabbitMQ и независимые ML-воркеры.

Nginx дополнительно используется как единая точка входа.

## 12. Ограничения MVP

* регистрация, авторизация и роли;
* загрузка произвольных пользовательских файлов;
* облачное развёртывание и Kubernetes;

* drift monitoring;
* Prometheus и Grafana;
* автоматические retries и dead-letter queue;
* хранение всех входных признаков в БД.
