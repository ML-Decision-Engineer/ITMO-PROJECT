**Датасет для решения задачи прогнозирования дефектов продукции**

Датасет содержит исторические производственные данные по рулонам и листам металлургического производства.  
Внутренний заказчик и производственная площадка не раскрываются по требованиям руководства.

Цель датасета — подготовить данные для ML-задачи раннего прогнозирования дефекта продукции до получения лабораторного факта.

**Исходный датасет:**

- 252 609 строк;
- 2 895 колонок;
- период данных: с 2025-06-01 по 2026-04-30;

- 26 854 строки;

Для рабочей постановки оставлены только релевантные строки:

- следующий цех — ЦЕХ-2;
- конечный цех — ЦЕХ-2;
- тип продукта — ПРОДУКТ г/к;

**Состав колонок:**

- float64: 2 828 колонок;

## 3. Оценка качества разметки

`notna_means_positive`

Разметка сильно несбалансирована: дефект является редким событием. Поэтому Accuracy не подходит как основная метрика. Для оценки качества модели нужны:

- Precision;
- Recall;
- PR-AUC;
- Recall@Top-K;
- Precision@Top-K.

Также есть особенность бизнес-процесса: лабораторный факт появляется с задержкой. С учётом задержки около 14 дней строки после 2026-04-16 требуют дополнительной проверки зрелости таргета. В последнем окне задержки найдено 1 146 потенциально незрелых строк - из анализа удалены.

Дополнительно найдены конфликтующие метки по возможным ID/key-колонкам. Это не означает автоматическую ошибку, но требует проверки бизнес-логики: один и тот же идентификатор может относиться к разным этапам, версиям или связанным производственным объектам.

Внтури производства данные собраны по множеству таблиц поэтому есть пересекающиеся ключи.


## 4. Важные признаки для моделирования

Важными для моделирования считаются только признаки, которые доступны на момент прогноза и не содержат лабораторный факт или результат контроля.

**Кандидаты в признаки:**

- технологические параметры производства;
- параметры заказа и продукта, известные до проверки;
- агрегированные производственные признаки;
- исторические признаки, рассчитанные только из прошлого;
- категориальные параметры продукта и процесса.

**Признаки, которые нельзя использовать напрямую:**

- ID и служебные ключи;
- даты и поля, используемые только для разделения выборок;
- признаки с результатами лаборатории;
- признаки с решениями по карантину;
- признаки с фактом дефекта;
- post-fact признаки;
- признаки с потенциальной утечкой из будущего;
- константные и почти пустые признаки.

Итого, важность признаков должна определяться не только статистикой, но и моментом доступности признака в реальном производственном процессе.


## 5. Baseline

Базовая частота дефекта в рабочей выборке:

**7.05%**

Это означает:

Базовая доля дефекта в рабочей выборке составляет 7.05%: 

1 892 дефектных объекта из 26 854. 

Поэтому случайный baseline по Precision равен примерно 7.05%.

All-positive baseline находит все дефекты и даёт Recall = 1.0, 

но его Precision также равен только 7.05%, 

потому что дефектом помечается вся продукция. 

All-negative baseline не находит дефекты и даёт Recall = 0.

Для top-10% рискованной продукции размер списка составляет примерно 2 685 объектов. 

При случайном выборе в такой список ожидаемо попадёт около 10% всех дефектов, то есть примерно 189 дефектных объектов. 

Поэтому хорошая модель должна концентрировать в top-risk зоне больше дефектов, чем случайный отбор.


## 6. Алгоритм формирования выборки

1. Взять исторические строки продукции с доступным лабораторным фактом.
2. Применить доменные фильтры рабочей постановки:
   - ЦЕХ-2 как следующий цех;
   - ЦЕХ-2 как конечный цех;
   - тип продукта — ПРОДУКТ г/к;
   - исключение нерелевантных назначений.
3. Сформировать бинарный таргет:
   - 1 — есть запись в целевой колонке карантина;
   - 0 — записи нет.
4. Исключить из признаков:
   - таргет;
   - даты;
   - ID;
   - служебные поля;
   - признаки с утечкой;
   - post-fact признаки;
   - константные и почти пустые признаки.
5. Обработать пропуски и типы данных.
6. Отдельно сохранить ID и дату для контроля качества, трассировки и backtest.
7. Разделить данные по времени.
8. Проверять качество не только по строкам, но и по дневной агрегации процента дефекта.



## 7. Стратегия валидации

Для задачи выбрана временная валидация, потому что продукт должен прогнозировать будущие производственные дни. Случайный split в такой задаче может дать завышенную оценку качества из-за утечек времени.

Используется схема:

- train — старые данные для обучения;
- validation — более новые данные для подбора порога;
- pretest — дополнительная проверка перед финальным тестом;
- holdout — последний временной период как честная имитация будущего.

**Распределение по выборкам:**

- train: 15 954 строки, positive_rate = 7.33%;
- validation: 3 191 строка, positive_rate = 7.93%;
- pretest: 2 128 строк, positive_rate = 8.41%;
- holdout: 5 581 строка, positive_rate = 5.20%.

holdout - сделан больше по размеру т.к иначе примерно в 2500 строк - дефекта всего 2.3% 

Этот отрезок времени на производстве был менее дефектным.

## 8. Оценка качества данных и предложения по улучшению

**Основные проблемы данных:**

- сильный дисбаланс классов;
- большое количество колонок с пропусками;
- большое количество константных и почти константных признаков;
- много потенциальных post-fact / leakage колонок;
- задержка появления таргета;
- возможные конфликтующие метки по ID/key-колонкам;
- нестабильность дневной доли дефекта по времени.

**Что нужно улучшить:**

1. Зафиксировать data contract:
   - какие признаки доступны на момент прогноза;
   - кто владелец данных;
   - как часто обновляются данные;
   - в каком формате приходят признаки.

2. Проверить все признаки-кандидаты на утечки:
   - лабораторные результаты;
   - карантинные решения;
   - фактические дефекты;
   - признаки, появляющиеся после контроля.

3. Проверить зрелость таргета:
   - не использовать последние дни, если лабораторный факт ещё не успел появиться.

4. Согласовать признаки с экспертами производства:
   - какие признаки реально доступны утром текущего дня;
   - какие признаки появляются только после обработки или лаборатории.

5. Для моделирования использовать метрики, устойчивые к дисбалансу:
   - Precision;
   - Recall;
   - PR-AUC;
   - Precision@Top-K;
   - Recall@Top-K.

6. Для бизнес-оценки отдельно считать:
   - ошибку дневного процента дефекта;
   - ошибку недельного процента дефекта;
   - стабильность прогноза по производственным дням.

## 9. Итог

Датасет подходит для решения задачи, но требует строгой подготовки. Основная сложность — не только в обучении модели, а в корректной работе с данными: нужно убрать утечки, учесть задержку таргета, обработать дисбаланс и построить честную временную валидацию.

Финальная постановка для моделирования:

**по историческим производственным данным обучить модель, которая заранее оценивает риск дефекта по каждой единице продукции и позволяет рассчитать ожидаемый дневной процент дефекта до получения лабораторного факта.**

Главный принцип подготовки данных:

**использовать только те признаки, которые доступны на момент реального прогноза, и проверять качество на будущих временных периодах.**
