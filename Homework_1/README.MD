# Бизнес-анализ проекта ML-сервиса прогнозирования дефектов продукции

## 1. Бизнес-анализ (Business Understanding)

Проект направлен на раннее прогнозирование дефектов продукции.  
Фактический результат контроля становится известен только через 10-14 дней, поэтому производство узнаёт о проблемах постфактум.

Цель проекта — создать ML-сервис, который по данным текущего дня оценивает риск дефекта по каждой единице продукции и прогнозирует дневной процент дефекта. Это позволит раньше реагировать на отклонения и снижать потери от брака, карантинов и претензий.

**Основные пользователи результата:**
- отдел качества;
- технологи производства;

**Бизнес-ценность проекта:**
- раннее обнаружение риска дефектов;
- снижение доли продукции, уходящей в карантин или брак;
- более точное планирование лабораторного контроля;
- снижение потерь от позднего обнаружения проблем;

---

## 1.1. Текущая ситуация (Assessing Current Solution)

Доступен исторический датасет:

- более **190 000 строк** уникальных продуктов;
- около **3000 признаков** включая смежные производственные линии;
- около **60 свойств/таргетов**;
- для первого этапа выбрано **5 ключевых свойств**;
- базовый MVP начинается с одного свойства — **Карантины | Свойство_1**;
- положительный класс сильно несбалансирован: дефектных случаев менее **8%**.

**Главная особенность — задержка таргета.**  
Признаки текущего дня уже доступны, а факт дефекта появляется только через 10-14 дней. Поэтому нужен ранний прогноз до лабораторной разметки.

---

### Риски проекта

**Сильный дисбаланс классов.** Дефектов менее 8%, поэтому используем PR-AUC, Precision/Recall, подбор порогов, class weights и калибровку вероятностей.

**Низкое качество данных.** Пропуски, шум и ошибки в таргетах снижают качество модели, поэтому нужны очистка данных, анализ пропусков и согласование признаков с экспертами.

**Задержка таргета.** Факт дефекта приходит через 10-14 дней, поэтому качество проверяется на исторических данных через backtest по производственным дням.

**Утечки данных.** В признаки могут попасть данные из будущего, поэтому нужен строгий временной split и проверка момента доступности каждого признака.

**Ошибка дневного процента.** Модель может хорошо работать на строках, но ошибаться в дневной агрегации, поэтому отдельно оцениваются качество классификации и ошибка дневного процента.


## 1.2. Решаемые задачи с точки зрения аналитики (Data Mining Goals)

Проект решает две аналитические задачи.

### Уровень 1. Прогноз дефекта по отдельной строке продукции

**Целевые метрики от производства:**

На первом этапе решается бинарная классификация для свойства **Карантины | Свойство_1**: модель оценивает вероятность дефекта для каждой единицы продукции.

| Метрика   | Минимальный целевой уровень |
| --------- | --------------------------- |
| Precision | около 65%                   |
| Recall    | не ниже 40%                 |


### Уровень 2. Прогноз дневного процента дефекта

По вероятностям строк рассчитывается прогноз дневного процента дефекта. Факт появляется позже, после лабораторной проверки.

**Критерии успешности:**

| Показатель                 | Целевой уровень                         |
| -------------------------- | --------------------------------------- |
| Ошибка дневного процента   | не более 5 процентных пунктов за день   |
| Ошибка недельного процента | не более 2 процентных пунктов за неделю |

## 1.3. План проекта (Project Plan)

### 1. Business Understanding

Цель этапа — согласовать бизнес-задачу, пользователей, ограничения и критерии успеха.


### 2. Data Understanding

Цель этапа — изучить данные и проверить их пригодность для моделирования.

**Основные действия:**

* изучить 190+ тыс. строк и 3000 признаков;
* проверить 60 свойств/таргетов;
* отдельно проанализировать 5 выбранных свойств;
* оценить дисбаланс классов;
* проверить пропуски, выбросы, дубликаты;
* определить признаки, доступные на момент прогноза;
* исключить признаки с потенциальной утечкой из будущего.

### 3. Data Preparation

Цель этапа — подготовить данные для обучения моделей.

Основные действия:

* очистка данных;
* обработка пропусков;
* формирование временного train/validation/test split;
* подготовка таргета для первого свойства;
* подготовка агрегатов по дням;
* проверка, что в обучении не используются будущие данные.


### 4. Modeling

Цель этапа — построить модель прогнозирования дефекта.

Базовая постановка:

* бинарная классификация для свойства: Карантины | Свойство_1;
* прогноз вероятности дефекта для каждой строки;
* подбор порога под требования Precision/Recall.

### 5. Evaluation

Цель этапа — проверить качество модели на уровне строк и на уровне дней.

Оценка проводится на двух уровнях:

* по строкам продукции: Precision, Recall, F1-score, PR-AUC;
* по дням: ошибка дневного процента и ошибка недельного агрегата.

### 6. Deployment

Цель этапа — подготовить модель к использованию в производственном контуре.

**Пилотный сценарий внедрения:**

1. Загружаются данные текущей продукции.
2. Модель рассчитывает вероятность дефекта и дневной процент.
3. Пользователи получают отчёт или дашборд с рискованными позициями.
4. Через 10-14 дней прогноз сравнивается с лабораторным фактом.
5. Результаты используются для мониторинга и дообучения модели.
