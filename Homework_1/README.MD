
# Домашнее задание №1  
# Бизнес-анализ проекта ML-сервиса прогнозирования дефектов продукции

## 1. Бизнес-анализ (Business Understanding)

Проект направлен на раннее прогнозирование дефектов продукции на производстве.  
Сейчас фактический результат контроля становится известен только через 10-14 дней, когда лаборатория завершает проверку. Из-за этого производство узнаёт о проблемах постфактум: когда продукция уже выпущена, часть решений уже принята, а риск брака или карантина мог быть высоким ещё в день производства.

Цель проекта — создать ML-сервис, который по данным текущего производственного дня заранее оценивает риск дефекта для каждой единицы продукции, а затем рассчитывает ожидаемый дневной процент дефекта. Это позволит производству раньше видеть потенциальные проблемы качества, быстрее реагировать на отклонения и снижать потери от брака, карантинов и претензий заказчиков.

**Проект предлагается развивать поэтапно:**

1. Сначала спрогнозировать один конкретный дефект/свойство.
2. Затем по вероятностям модели рассчитать дневной процент этого дефекта.
3. После этого расширить подход на несколько свойств и прогнозировать общий дневной процент дефектной продукции.

В качестве первого свойства для прототипа выбрано **Карантины | Свойство_1**. Далее заводом планируется использовать 5 наиболее важных свойств из 60 доступных.

**Основные пользователи результата:**
- отдел качества;
- технологи производства;
- руководители смен и производственных участков;
- аналитики и ML-команда;
- менеджеры, отвечающие за выполнение заказов и работу с клиентами.

**Бизнес-ценность проекта:**
- раннее обнаружение риска дефектов;
- снижение доли продукции, уходящей в карантин или брак;
- более точное планирование лабораторного контроля;
- снижение потерь от позднего обнаружения проблем;
- повышение доверия заказчиков за счёт стабильного качества;
- переход от реактивного контроля качества к проактивному управлению.

## 1.1. Текущая ситуация (Assessing Current Solution)

На текущем этапе производство располагает историческим датасетом:

- более **190 000 строк** уникальных продуктов;
- около **3000 признаков** включая смежные производственные линии;
- около **60 свойств/таргетов**, по которым можно прогнозировать дефекты;
- для первого этапа выбрано **5 ключевых свойств**;
- базовый MVP начинается с одного свойства — **Карантины | Свойство_1**;
- положительный класс сильно несбалансирован: дефектных случаев менее **8%**.

**Главная особенность задачи — задержка фактического таргета.**  
Для продукции текущего дня признаки уже доступны, но факт дефекта становится известен только спустя 10-14 дней после лабораторной проверки. Поэтому сервис должен давать ранний прогноз до появления фактической разметки.

**Текущая логика решения:**

```text
Строка продукции текущего дня → модель → вероятность дефекта p
````

Далее вероятности агрегируются по дню:

```text
Среднее значение вероятностей за день → прогноз дневного процента дефекта
```

**То есть сервис решает две связанные задачи:**

1. На уровне отдельной единицы продукции — определить вероятность дефекта.
2. На уровне дня — оценить ожидаемый процент дефектной продукции.

---

### Риски проекта

**Риски проекта и план их снижения**

**Сильный дисбаланс классов.**
Дефектов в данных менее 8%, поэтому модель может плохо находить редкие случаи. Для снижения риска нужно использовать метрики PR-AUC, Precision и Recall, подбирать пороги классификации, применять class weights и калибровку вероятностей.

**Низкое качество данных.**
Пропуски, шум, ошибки в признаках и таргетах могут снизить качество модели. Для снижения риска необходимо провести очистку данных, анализ пропусков, проверку утечек и согласовать ключевые признаки с экспертами производства.

**Задержка таргета.**
Факт дефекта становится известен только через 10-14 дней после лабораторной проверки. Для снижения риска модель нужно обучать на исторических данных и проверять качество через backtest по производственным дням.

**Утечки данных.**
В признаки могут попасть данные, которые становятся известны только после лабораторной проверки. Для снижения риска необходимо использовать строгий временной split и проверять момент доступности каждого признака в отдельности.

**Недостаточная точность дневного процента.**
Модель может хорошо классифицировать отдельные строки, но ошибаться при расчёте дневного процента дефекта. Для снижения риска нужно отдельно оценивать качество классификации и качество дневной агрегации.

**Недоверие со стороны производства.**
Пользователи могут не принять модель, если её прогнозы будут непонятны. Для снижения риска нужно добавить интерпретацию факторов риска и сделать понятные отчёты или дашборды для технологов и отдела качества.

**Сложность масштабирования на 5+ свойств.**
Разные свойства могут иметь разную частоту дефектов и разное качество моделей. Для снижения риска проект нужно запускать поэтапно: сначала одно свойство, затем пять свойств, затем общий процент дефектной продукции.


## 1.2. Решаемые задачи с точки зрения аналитики (Data Mining Goals)

С аналитической точки зрения проект состоит из двух уровней.

### Уровень 1. Прогноз дефекта по отдельной строке продукции

Для каждой единицы продукции модель должна предсказать вероятность дефекта:

```text
p = P(дефект = 1)
```

На первом этапе решается бинарная классификация для свойства **Карантины | Свойство_1**:

```text
0 — дефекта нет
1 — дефект есть
```

После успешного прототипа подход расширяется на 5 выбранных свойств.

**Целевые метрики от производства:**

| Метрика   | Минимальный целевой уровень |
| --------- | --------------------------- |
| Precision | около 65%                   |
| Recall    | не ниже 40%                 |

Precision важен, чтобы модель не создавала слишком много ложных тревог.
Recall важен, чтобы модель находила значимую часть реальных дефектов и не пропускала критичные случаи.

Так как классы сильно несбалансированы, Accuracy не является основной метрикой. Основной фокус — на Precision, Recall, F1-score и PR-AUC.

### Уровень 2. Прогноз дневного процента дефекта

После получения вероятностей по всем строкам текущего дня рассчитывается прогноз дневного процента.

Фактический дневной процент после получения лабораторных результатов.

Задача модели — заранее оценить дневной процент дефекта до появления факта.

**Критерии успешности:**

| Показатель                 | Целевой уровень                         |
| -------------------------- | --------------------------------------- |
| Ошибка дневного процента   | не более 5 процентных пунктов за день   |
| Ошибка недельного процента | не более 2 процентных пунктов за неделю |

Это важно, потому что производству нужен не только прогноз по отдельным единицам продукции, но и общий индикатор качества дня и недели.

### Расширение на несколько свойств

После первого свойства модельный контур расширяется на 5 выбранных свойств.
Для каждого свойства прогнозируется отдельная вероятность дефекта.

Далее можно считать:

1. Дневной процент по каждому свойству отдельно.
2. Общий дневной процент продукции, у которой ожидается хотя бы один дефект.

Для общего процента нельзя просто сложить проценты по свойствам, потому что одна единица продукции может иметь риск сразу по нескольким дефектам. Поэтому требуется отдельная логика агрегации вероятностей на уровне строки, а затем расчёт процента на уровне дня.

## 1.3. План проекта (Project Plan)

Проект строится по стандартному итеративному процессу разработки ML-сервиса и включает 6 фаз.

### 1. Business Understanding

Цель этапа — согласовать бизнес-задачу, пользователей, ограничения и критерии успеха.

Результаты этапа:

* выбрана основная бизнес-цель: ранний прогноз дефектов и дневного процента брака;
* определён первый MVP дефект: Карантины | Свойство_1;
* зафиксированы целевые метрики: Precision около 65%, Recall не ниже 40%;
* определены требования к дневному и недельному прогнозу процента дефекта;
* описаны риски проекта.

### 2. Data Understanding

Цель этапа — изучить данные и проверить их пригодность для моделирования.

**Основные действия:**

* изучить 190+ тыс. строк и 3000 признаков;
* проверить 60 свойств/таргетов;
* отдельно проанализировать 5 выбранных свойств;
* оценить дисбаланс классов;
* проверить пропуски, выбросы, дубликаты;
* определить признаки, доступные на момент прогноза;
* исключить признаки с потенциальной утечкой из будущего.

Результат этапа — понимание структуры данных, качества таргетов и ограничений для честного прогноза.

### 3. Data Preparation

Цель этапа — подготовить данные для обучения моделей.

Основные действия:

* очистка данных;
* обработка пропусков;
* кодирование категориальных признаков;
* нормализация или отбор признаков при необходимости;
* формирование временного train/validation/test split;
* подготовка таргета для первого свойства;
* подготовка агрегатов по дням;
* проверка, что в обучении не используются будущие данные.

Результат этапа — готовая обучающая выборка для первой модели и дневной агрегации.

### 4. Modeling

Цель этапа — построить модель прогнозирования дефекта.

Базовая постановка:

* бинарная классификация для свойства: Карантины | Свойство_1;
* прогноз вероятности дефекта для каждой строки;
* подбор порога под требования Precision/Recall.

Возможные модели:

* Logistic Regression как простой baseline;
* Random Forest;
* CatBoost / LightGBM / XGBoost как основные модели для табличных данных;
* калибровка вероятностей для корректного дневного процента.

Результат этапа — модель, которая выдаёт вероятность дефекта по каждой единице продукции.

### 5. Evaluation

Цель этапа — проверить качество модели на уровне строк и на уровне дней.

Оценка на уровне строк:

* Precision;
* Recall;
* F1-score;
* PR-AUC;
* матрица ошибок.

Оценка на уровне дней:

* ошибка дневного процента;
* средняя абсолютная ошибка в процентных пунктах;
* ошибка недельного агрегата;
* сравнение прогнозного и фактического процента по дням.

Критерий успешности:

* Precision около 65%;
* Recall не ниже 40%;
* ошибка дневного процента не более 5 п.п.;
* ошибка недельного процента не более 2 п.п.

Если модель выполняет эти требования, её можно считать пригодной для пилотного использования.

### 6. Deployment

Цель этапа — подготовить модель к использованию в производственном контуре.

**Пилотный сценарий внедрения:**

1. Каждый день загружаются данные текущей продукции.
2. Модель рассчитывает вероятность дефекта для каждой строки.
3. Вероятности агрегируются в дневной процент.
4. Пользователи получают отчёт или дашборд:

   * риск по каждой единице продукции;
   * прогноз дневного процента дефекта;
   * список наиболее рискованных позиций;
   * сравнение с историческими значениями.
5. Через 10-14 дней, когда приходит лабораторный факт, прогноз сравнивается с реальностью.
6. Результаты используются для мониторинга качества модели и последующего дообучения.

Итоговая целевая система должна помогать производству заранее видеть риск дефектов, точнее планировать контроль качества и быстрее реагировать на возможные проблемы.
