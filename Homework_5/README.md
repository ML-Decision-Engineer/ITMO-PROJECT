## 3. Выбор ML-метрик

Для задачи классификации будут использоваться:

* Precision;
* Recall;
* F1-score;
* PR-AUC;
* Precision@Top-K;
* Recall@Top-K.

Главная бизнес-логика:

* Precision важен, потому что ложные тревоги создают лишнюю проверку и нагрузку на производство;
* Recall важен, потому что модель не должна пропускать слишком много дефектов;
* PR-AUC важен из-за сильного дисбаланса классов;
* Top-K метрики важны, потому что производству нужен ограниченный список наиболее рискованной продукции.

### Дополнительное требование из задания

Нужно использовать не менее двух метрик качества и дополнительно оценить совокупную метрику качества.

Поэтому будет рассчитана не только отдельная ML-метрика, но и общий итоговый score, который учитывает несколько сторон качества модели.

Пример логики совокупной оценки:

* модель должна находить дефекты;
* модель не должна давать слишком много ложных тревог;
* модель должна концентрировать дефекты в top-risk зоне;
* модель должна сохранять качество на holdout.

---

## 4. Связь ML-метрик с продуктовой задачей

ML-метрики нужны не сами по себе, а для поддержки производственного решения.

Связь такая:

* Precision отвечает за качество списка продукции, которую модель пометила как рискованную;
* Recall показывает, какую долю всех дефектов модель смогла найти;
* Recall@Top-K показывает, сколько дефектов попадёт в ограниченную зону дополнительной проверки;
* ошибка дневного процента дефекта показывает, насколько прогноз полезен для управления качеством по дням.

### Рекомендация по выполнению

Отдельно нужно показать, что хорошая ML-модель не обязательно автоматически полезна для бизнеса.
Поэтому после оценки классификации нужно сделать дневную агрегацию и проверить, насколько прогноз модели помогает оценивать дневной процент дефекта.

---

## 5. Baseline: что будет выбрано

Baseline — это простое решение, с которым будут сравниваться все дальнейшие модели.

Планируется проверить несколько baseline-подходов.

---

### 5.1. Наивный baseline

* all-negative: считать всю продукцию недефектной;
* all-positive: считать всю продукцию дефектной;
* random / frequency baseline: прогнозировать риск на уровне средней частоты дефекта.

Задача этого блока — показать минимальный уровень качества, ниже которого модель не имеет смысла.

---

### 5.2. Top-K baseline

Будет проверяться, сколько дефектов можно найти, если случайно выбрать top-10% продукции.

Это нужно, чтобы сравнить будущую модель с простым случайным отбором.

---

### 5.3. Простые ML-модели

После наивных baseline будут построены 1–2 простые модели:

* Logistic Regression;
* Decision Tree;
* Random Forest или HistGradientBoosting.

На этом этапе задача не в максимальном качестве, а в том, чтобы получить первую честную точку сравнения.

---

### 5.4. Усложнённая модель

Дополнительно будет построена одна более сложная модель:

* CatBoost / LightGBM;
* либо модель с подбором гиперпараметров;
* либо AutoML-подход.

Усложнённая модель нужна, чтобы проверить, даёт ли более сильный алгоритм реальный прирост качества по сравнению с простыми baseline.

---

### 5.5. AutoML

AutoML будет использоваться как дополнительный эксперимент, а не как замена логике анализа.

План:

* задать тот же train / validation / holdout split;
* не перемешивать данные случайно;
* ограничить время AutoML-обучения;
* сравнить AutoML с простыми моделями по тем же метрикам;
* не подбирать решение по holdout;
* использовать holdout только для финальной честной проверки.

Главная идея: AutoML может помочь найти сильную модель, но качество всё равно оценивается по честной временной валидации и продуктовым метрикам.

---

## 6. Стратегия валидации

Будет использоваться временное разделение данных:

* train — старые данные;
* validation — более новые данные для подбора порога;
* pretest — дополнительная проверка;
* holdout — последний период как имитация будущих производственных дней.

Random split использовать нельзя, потому что он может завысить качество из-за временных утечек.

### Рекомендация по выполнению

Все модели должны сравниваться на одинаковом разбиении данных.
Порог классификации лучше подбирать на validation, затем проверять его на pretest и holdout.

Holdout нельзя использовать для выбора модели много раз, иначе он перестанет быть честной имитацией будущего.

---

## 7. Оценка baseline и моделей

Для каждого baseline и каждой модели будут рассчитаны:

* Precision;
* Recall;
* F1-score;
* PR-AUC;
* Precision@Top-10%;
* Recall@Top-10%;
* количество объектов, попавших в риск-зону;
* доля найденных дефектов в риск-зоне.

Отдельно будет проверяться дневная агрегация:

* прогнозный дневной процент дефекта;
* фактический дневной процент дефекта;
* ошибка дневного процента;
* ошибка недельного процента.

### Совокупная метрика качества

Дополнительно будет рассчитана итоговая сравнительная оценка модели, которая объединяет несколько критериев:

* качество поиска дефектов;
* качество top-risk зоны;
* стабильность на holdout;
* ошибку дневного прогноза.

Это нужно, чтобы выбрать модель не по одной метрике, а по совокупной полезности для задачи.

---

## 8. Что будет считаться улучшением baseline

Модель считается полезной, если она:

* превосходит случайный baseline по Precision;
* превосходит случайный baseline по Recall@Top-K;
* концентрирует больше дефектов в top-risk зоне;
* сохраняет качество на holdout;
* даёт осмысленный прогноз дневного процента дефекта;
* показывает улучшение по совокупной метрике качества.

---

## 9. Что делать, если сложная модель не бьёт baseline

Если более сложная модель или AutoML не улучшает baseline, нужно проверить:

* нет ли утечек или ошибок в подготовке данных;
* правильно ли выбрана метрика;
* хватает ли данных для выбранной постановки;
* не слишком ли шумный таргет;
* не является ли простая модель достаточной для текущего этапа;
* нужно ли улучшать данные, а не модель;
* не переобучилась ли сложная модель на train / validation.

Важно: если AutoML не даёт прироста, это тоже полезный результат. Это означает, что проблема может быть не в алгоритме, а в данных, таргете, признаках или постановке задачи.

9. Оценить качество не только по ML-метрикам, но и по дневной продуктовой агрегации.
